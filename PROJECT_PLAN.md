# Windows 桌面小怪兽项目规划

## 1. 项目愿景

这个项目想做的是一个常驻 Windows 桌面的小精灵。它的灵感来自“住在 macOS 菜单栏里的小怪兽”，但目标平台是 Windows，所以不应该强行照搬 Mac 菜单栏，而应该改成更符合 Windows 的形态：

- 用系统托盘图标负责常驻、菜单和基础控制。
- 用一个透明的桌面悬浮小窗展示小怪兽本体。
- 根据 CPU 负载、空闲时间、打字频率改变小怪兽的心情。
- 支持“投喂”玩法：把文件拖进投喂窗口，小怪兽做吞噬动画。

核心目标不是做一个复杂工具，而是做一个低存在感但有灵性的桌面小玩具。它大多数时候不打扰你，但偶尔会让你觉得“它真的在陪我干活”。

## 2. 目标平台

主要目标平台：

- Windows 10
- Windows 11

当前开发环境：

- 你现在用的是 macOS。
- 大部分 UI、状态机、动画和项目结构都可以在 macOS 上开发。
- Windows 安装包可以通过 GitHub Actions 在云端构建。
- 最终体验仍然需要在真实 Windows 机器上测试。

第一版暂时不做：

- macOS 菜单栏版本。
- Linux 版本。
- 复杂 AI 行为。
- 读取或保存用户具体输入内容。

## 3. 推荐技术栈

推荐使用：

- Tauri 2
- React
- Vite
- TypeScript
- Rust

为什么推荐这套：

- Tauri 比 Electron 更轻，更适合做常驻后台的小应用。
- React + TypeScript 适合快速写界面、状态机和动画逻辑。
- Rust 适合处理系统层能力，比如 CPU 采样、空闲检测、托盘菜单、Windows API 调用。
- 可以通过 GitHub Actions 构建 Windows 版本，不需要你本地装虚拟机。

备选方案：

- Electron + React：开发最快、资料最多，但运行时更重。
- Python + PySide6 + pystray + psutil：适合快速做原型，但透明窗口、动画表现和 Windows 打包会更麻烦。

当前建议：

- 正式项目使用 Tauri。
- 如果之后想先做一个很粗糙的验证版，可以考虑 Python。

## 4. 产品形态

这个应用主要有三个可见部分：

1. 系统托盘图标
2. 透明悬浮小怪兽窗口
3. 投喂窗口和设置窗口

托盘图标负责：

- 显示或隐藏小怪兽
- 打开投喂功能
- 开启或关闭安静模式
- 打开设置
- 查看当前心情
- 退出应用

悬浮小怪兽窗口负责：

- 显示小怪兽动画
- 使用透明背景
- 没有普通窗口边框
- 可以置顶
- 可以拖动位置
- 可以开启鼠标穿透，让它不影响正常操作

投喂窗口负责：

- 出现一个临时文件拖拽区域
- 用户把文件拖进去
- 应用读取文件的基础信息
- 小怪兽播放吞噬动画
- 根据文件类型给出不同反应

## 5. 核心玩法

### 5.1 根据 CPU 负载改变心情

小怪兽可以根据 CPU 使用率改变状态：

- CPU 低：悠闲、发呆
- CPU 中等：认真、警觉
- CPU 高：流汗、紧张
- CPU 持续很高：崩溃、求饶、冒烟

注意：

- 不应该直接用某一秒的 CPU 数值决定状态。
- 应该做平滑处理，避免 CPU 突然跳一下，小怪兽就疯狂切换状态。

建议阈值：

- `0-35%`：平静
- `35-70%`：专注
- `70-90%`：压力大
- `90% 以上持续 10 秒`：过载

### 5.2 根据打字频率改变心情

小怪兽可以根据打字速度反应：

- 正常打字：认真陪你工作
- 快速打字：有点兴奋
- 非常快速打字：满头大汗，求你慢点
- 长时间不打字：开始犯困

隐私原则：

- 只统计按键事件数量。
- 不记录按了什么键。
- 不保存用户输入内容。
- 不做任何键盘记录器行为。

### 5.3 根据空闲时间睡觉

当用户长时间没有操作，小怪兽会进入睡觉流程：

- 短时间空闲：眨眼、看周围
- 中等空闲：打哈欠
- 长时间空闲：掏出小床睡觉
- 用户回来操作：醒来动画

建议阈值：

- `2 分钟`：犯困
- `5 分钟`：睡觉
- `30 分钟`：深度睡眠

### 5.4 文件投喂

用户可以把文件拖入投喂窗口，小怪兽做吞噬动作。

第一版只读取安全的文件元信息：

- 文件名
- 文件后缀
- 文件大小
- 修改时间

可以设计的反应：

- 代码文件：咔嚓咔嚓啃代码
- 图片文件：开心发光
- 压缩包：噎住或者艰难咀嚼
- 未知文件：疑惑地吃

第一版不读取文件内容，也不上传任何文件。

## 6. 小怪兽状态机

第一版可以先设计这些状态：

- `idle`：发呆
- `focused`：专注
- `stressed`：紧张
- `overheated`：过载
- `sleepy`：犯困
- `sleeping`：睡觉
- `eating`：进食
- `happy`：开心
- `sad`：难过
- `hidden`：隐藏

状态规则：

- `eating` 投喂状态优先级较高，应该临时覆盖大多数状态。
- `hidden` 隐藏状态不显示小怪兽，但后台可以继续采样，除非开启安静模式。
- `sleeping` 睡觉状态在用户恢复操作时立刻被打断。
- `overheated` 过载状态必须由持续高 CPU 或持续高速打字触发。
- 状态切换要有防抖，避免动画频繁跳来跳去。

简单状态流转示意：

```text
idle -> focused -> stressed -> overheated
idle -> sleepy -> sleeping
sleeping -> idle
任意可见状态 -> eating -> happy/idle
任意可见状态 -> hidden
hidden -> 上一个可见状态
```

## 7. 技术架构

### 7.1 前端部分

前端主要负责：

- 渲染小怪兽动画
- 管理 UI 状态
- 显示设置窗口
- 显示投喂窗口
- 把用户操作发送给后端
- 接收后端传来的系统数据

建议目录结构：

```text
src/
  app/
    App.tsx
    routes.ts
  pet/
    PetWindow.tsx
    petStateMachine.ts
    petAnimations.ts
    petTypes.ts
  feeding/
    FeedingWindow.tsx
    feedingTypes.ts
  settings/
    SettingsWindow.tsx
    settingsStore.ts
  bridge/
    tauriEvents.ts
    commands.ts
```

### 7.2 后端部分

后端主要负责：

- 创建系统托盘图标
- 管理应用窗口
- 采样 CPU 使用率
- 检测用户空闲时间
- 安全统计打字频率
- 保存设置
- 管理开机启动
- 给前端发送事件

建议目录结构：

```text
src-tauri/
  src/
    main.rs
    tray.rs
    windows.rs
    config.rs
    sensors/
      mod.rs
      cpu.rs
      idle.rs
      keyboard.rs
    pet/
      mod.rs
      mood.rs
```

### 7.3 数据流

```text
Windows 系统 API
  -> Rust 传感器模块
  -> 心情聚合逻辑
  -> Tauri 事件
  -> React 状态机
  -> 小怪兽动画渲染
```

## 8. Windows 专属设计注意点

### 8.1 托盘不是动画主体

Windows 的系统托盘图标很小，而且可能被系统自动收进隐藏区域。

所以托盘图标应该只负责控制，不应该承载完整动画。

真正的小怪兽应该放在透明悬浮窗口里。

### 8.2 透明悬浮窗是关键

小怪兽的主要体验来自一个透明小窗：

- 无边框
- 背景透明
- 可以置顶
- 可以拖动
- 可以设置鼠标穿透
- 可以贴近任务栏或屏幕角落

### 8.3 投喂不要直接拖到托盘

在 Windows 上，直接把文件拖到托盘图标不是一个稳定可靠的交互。

更好的方式是：

1. 用户点击托盘菜单里的“投喂”
2. 弹出小怪兽张嘴的投喂窗口
3. 用户把文件拖进窗口
4. 小怪兽播放吞噬动画

### 8.4 空闲检测

Windows 上可以用 `GetLastInputInfo` 获取用户最后一次输入时间。

这个能力适合判断用户是否离开电脑、是否该让小怪兽睡觉。

### 8.5 打字频率检测

可以用 Windows 键盘 hook 或 raw input 方案。

第一版只统计按键数量，不记录按键内容。

## 9. 隐私和安全边界

这个项目虽然是玩具，但它会接触系统活动，所以必须把隐私边界写清楚。

允许采集：

- CPU 使用率
- 用户空闲时长
- 最近一段时间的按键事件数量
- 用户主动投喂文件时的文件元信息

不允许采集：

- 用户具体输入内容
- 剪贴板内容
- 屏幕截图内容
- 文件内容
- 默认上传任何数据

必须提供的用户控制：

- 安静模式
- 暂停系统检测
- 隐藏小怪兽
- 关闭打字频率检测
- 清空本地统计数据

## 11. 构建和发布策略

开发阶段：

- 在 macOS 上开发前端、状态机、动画和通用逻辑。
- 在 macOS 上使用 mock 数据模拟 Windows 系统信息。
- Windows 专属能力放在平台隔离层里。

CI 构建：

- 使用 GitHub Actions 的 `windows-latest` 环境构建 Windows 产物。
- 同时跑 lint、测试和打包流程。

发布产物：

- 早期先发便携版 `.exe`。
- 稳定后再做安装包。

手工测试重点：

- Windows 10
- Windows 11
- 不同缩放比例
- 多显示器
- 睡眠和唤醒
- 开机启动
- 托盘隐藏区域
- 杀毒软件误报情况

## 12. 第一批开发任务

建议按这个顺序开工：

1. 初始化 Tauri + React + TypeScript 项目。
2. 添加系统托盘图标和菜单。
3. 创建透明、无边框的小怪兽窗口。
4. 实现前端小怪兽状态机。
5. 用 CSS 或图片序列做占位动画。
6. 添加模拟系统数据。
7. 添加 Rust CPU 采样模块。
8. 添加 Windows 空闲检测模块。
9. 添加安全打字频率统计模块。
10. 添加投喂窗口和文件拖拽。
11. 添加设置持久化。

## 13. 还需要确认的问题

这些问题可以等第一版骨架出来后再决定：

- 小怪兽默认贴在任务栏附近，还是默认自由拖动？
- 鼠标穿透默认开启还是关闭？
- 第一版画风用像素风、贴纸风，还是简单矢量风？
- 投喂功能是否永远只读文件元信息？
- 是否做一个本地统计页，比如今日摸鱼指数、专注时间、空闲时间？
- 小怪兽是否应该支持多套皮肤？

## 14. 当前推荐方案

第一版不要贪大，先做一个好玩的 MVP：

- 一个透明小怪兽窗口。
- 一个系统托盘图标。
- 三种心情：发呆、压力大、睡觉。
- 一个投喂动作。
- 不记录用户输入内容。
- 从第一天就准备 Windows CI 构建。

这样既能保留这个项目最有意思的地方，又不会一上来被系统 API、打包、动画资源和平台差异拖死。
